요약

문제: runtime.slam.mode: "offline" 으로 같은 시퀀스를 실행해도 매 실행마다 CameraTrajectoryTUM.txt가 달라졌다.

원인: Rerun SDK의 Tokio 백그라운드 I/O 스레드가 힙 메모리를 지속 할당·해제하면서 새로 생성되는 KeyFrame/MapPoint 객체의 주소를 매 실행마다 다르게 만든다. std::set<KeyFrame*>·std::map<KeyFrame*, …>는 포인터 주소 순서로 반복(iterate)되므로, 반복 순서 → g2o 엣지 삽입 순서 → BA 결과 → 궤적이 전부 달라진다.

해결: 모든 포인터 키 컨테이너의 비교자(Comparator)를 주소 비교에서 mnId 비교로 교체. 힙 레이아웃에 무관하게 항상 동일한 순서를 보장.

검증: rerun 활성화 상태에서 4회 연속 실행 → 모든 CameraTrajectoryTUM.txt 바이트 동일.

근본 원인: 포인터 주소 기반 컨테이너 순서

왜 포인터 주소가 달라지는가

glibc는 큰 힙 객체에 mmap을 사용하며, ASLR(Address Space Layout Randomization)과 tcache LIFO 재사용 때문에 동일한 프로그램이라도 실행마다 new KeyFrame()이 반환하는 주소가 달라진다.

오프라인 모드에서 Rerun SDK가 초기화되면 Tokio 비동기 런타임의 백그라운드 스레드가 시작되어 SLAM 처리 중에도 지속적으로 힙을 할당·해제한다. 이로 인해 KF/MP 객체가 받는 주소가 실행마다 다르게 된다.

포인터 순서가 결과에 영향을 미치는 경로

힙 레이아웃 변화
  → KeyFrame* / MapPoint* 주소 변화
  → std::set<KeyFrame*>, std::map<KeyFrame*, …> 반복 순서 변화
  → MapPoint::GetObservations() 반환값의 순서 변화
  → g2o LocalBundleAdjustment / GlobalBundleAdjustment 엣지 삽입 순서 변화
  → Hessian 행렬 누적 순서 변화 (부동소수점 덧셈은 비가환)
  → 최적화 결과 변화
  → CameraTrajectoryTUM.txt 변화

이 외에도 다음 경로에서 순서가 결과에 영향을 준다:

  • UpdateLocalKeyFrames()의 keyframeCounter 반복 → 참조 KF 선택 → 로컬 맵 구성 → 추적 정확도
  • GetChilds() 반복 + break → 첫 번째 자식 KF만 로컬 맵에 추가됨 → 로컬 맵이 달라짐
  • UpdateBestCovisibles() 정렬 타이브레이커 → 공변 그래프 순서 → 후속 알고리즘 입력
  • OptimizeEssentialGraph()의 LoopConnections 반복 → 루프 클로저 BA 결과

⚠ 진단 과정에서 발견한 중요한 사실

fprintf 진단 코드를 추가하면 힙 할당이 발생해 포인터 주소가 바뀌고, 비결정성의 위치 자체가 이동한다. 따라서 진단 코드를 넣은 상태의 비교 결과는 신뢰할 수 없다. 모든 진단 코드를 제거한 뒤에야 안정적인 비교가 가능했다.

Fix 1 — SLAMComparators.h (신규 파일)

1

include/SLAMComparators.h (신규)

포인터 비교자를 타입 레벨에서 교체하기 위한 핵심 헤더. KeyFrame·MapPoint를 전방선언(forward declaration)만 사용하여 순환 include 없이 양쪽 헤더에서 사용 가능.

+// include/SLAMComparators.h
+namespace ORB_SLAM2 {
+class KeyFrame;
+class MapPoint;
+
+struct KFIdLess {
+    bool operator()(const KeyFrame* a, const KeyFrame* b) const;
+};
+struct MPIdLess {
+    bool operator()(const MapPoint* a, const MapPoint* b) const;
+};
+} // namespace ORB_SLAM2

인라인 본체(body)는 전체 클래스 정의가 끝난 뒤 정의해야 mnId에 접근 가능:

  • KFIdLess::operator() 본체 → KeyFrame.h 닫는 괄호 바로 앞
  • MPIdLess::operator() 본체 → MapPoint.h 닫는 괄호 바로 앞
// KeyFrame.h 맨 끝 (namespace 닫힌 후)
inline bool ORB_SLAM2::KFIdLess::operator()(
    const ORB_SLAM2::KeyFrame* a, const ORB_SLAM2::KeyFrame* b) const {
    return a->mnId < b->mnId;
}

// MapPoint.h 맨 끝 (namespace 닫힌 후)
inline bool ORB_SLAM2::MPIdLess::operator()(
    const ORB_SLAM2::MapPoint* a, const ORB_SLAM2::MapPoint* b) const {
    return a->mnId < b->mnId;
}

왜 반환 타입도 바꿔야 하는가

반환 타입을 set<KeyFrame*>로 유지하면, 내부 멤버(set<KeyFrame*, KFIdLess>)를 복사할 때 비교자가 std::less<KeyFrame*>(포인터 주소 비교)로 바뀐다. 호출자가 받는 복사본이 다시 비결정적이 되므로 반환 타입도 반드시 교체해야 한다.

Fix 3 — g2o 엣지 삽입 순서 고정

3

Optimizer.cc — BundleAdjustment / LocalBundleAdjustment

g2o는 엣지를 삽입하는 순서대로 내부 ID를 부여하고, Hessian 행렬을 이 순서로 누적한다. 부동소수점 덧셈은 비가환(non-commutative)이므로 순서가 달라지면 결과도 달라진다.

Fix 2로 GetObservations()가 KF ID 순서의 맵을 반환하게 되어 이미 결정적이 되지만, 추가 안전장치로 명시적 정렬도 유지:

// Before: observations 반복 순서 = 포인터 주소 순서 (비결정적)
for(map<KeyFrame*,size_t>::const_iterator mit = observations.begin(); ...)

// After: KF ID 순서로 명시 정렬 후 엣지 삽입
// MODIFIED for "stable edge insertion order by KF id for deterministic offline g2o BA"
vector<pair<unsigned long, KeyFrame*>> vSortedObs;
for(const auto& kv : observations)
    vSortedObs.emplace_back(kv.first->mnId, kv.first);
sort(vSortedObs.begin(), vSortedObs.end());  // ID 오름차순

for(const auto& [kfId, pKFi] : vSortedObs)
    { ... optimizer.addEdge(e); ... }

OptimizeEssentialGraph의 LoopConnections 타입도 KFIdLess 적용:

- map<KeyFrame*, set<KeyFrame*>> LoopConnections;
+ map<KeyFrame*, set<KeyFrame*, KFIdLess>, KFIdLess> LoopConnections;

Fix 4 — 공변 그래프 정렬 및 참조 KF 타이브레이커

4

KeyFrame::UpdateBestCovisibles() & UpdateConnections()

코비전(공변) 그래프를 가중치 내림차순으로 정렬할 때, 가중치가 같은 KF끼리의 순서가 포인터 주소에 따라 달라졌다. KF ID를 타이브레이커로 추가:

- sort(vPairs.begin(), vPairs.end());
+ // MODIFIED for "stable covisibility sort by id tiebreaker for deterministic offline mode"
+ sort(vPairs.begin(), vPairs.end(),
+      [](const pair<int,KeyFrame*>& a, const pair<int,KeyFrame*>& b){
+          if(a.first != b.first) return a.first < b.first;
+          return a.second->mnId < b.second->mnId;
+      });

Tracking::UpdateLocalKeyFrames() — 참조 KF 선택

가장 많은 맵 포인트를 공유하는 KF를 참조 KF로 선택할 때, 공유 수가 같으면 ID가 낮은 KF를 일관되게 선택:

- if(it->second > max)
+ // MODIFIED for "stable reference keyframe selection by id tiebreaker for deterministic offline mode"
+ if(it->second > max ||
+    (it->second == max && pKFmax && pKF->mnId < pKFmax->mnId))

또한 로컬 KF 목록을 ID 순으로 정렬한 후 이웃 KF를 탐색하여, 코비전 이웃·자식·부모를 추가하는 순서가 결정적이 되도록 함:

+ // MODIFIED for "stable local keyframe ordering by id for deterministic offline mode"
+ sort(mvpLocalKeyFrames.begin(), mvpLocalKeyFrames.end(),
+      [](const KeyFrame* a, const KeyFrame* b){ return a->mnId < b->mnId; });

Fix 5 — SetBadFlag 자식 처리 순서 고정

5

KeyFrame::SetBadFlag() — 스패닝 트리 재구성

KF가 Bad로 표시될 때 자식 KF들의 새 부모를 찾는 루프가 mspChildrens(포인터 주소 순서 set)를 그대로 반복했다. KF ID 순으로 정렬한 벡터를 사용하고, 동점 시 ID 타이브레이커 적용:

- for(set<KeyFrame*>::iterator sit = mspChildrens.begin(); ...)
+ // MODIFIED for "stable child ordering by id when reassigning parents for deterministic offline mode"
+ vector<KeyFrame*> vChilds(mspChildrens.begin(), mspChildrens.end());
+ sort(vChilds.begin(), vChilds.end(), [](KeyFrame* a, KeyFrame* b){ return a->mnId < b->mnId; });
+
+ for(KeyFrame* pKF : vChilds) {
+     ...
+     int w = pKF->GetWeight(vpConnected[i]);
+     if(w > max || (w == max && pC != nullptr && pKF->mnId < pC->mnId)) {
+         pC = pKF; pP = vpConnected[i]; max = w; bContinue = true;
+     }
+ }

Fix 2로 mspChildrens가 set<KeyFrame*, KFIdLess>로 바뀌어 이 명시 정렬은 이제 중복이지만, 이전에 추가된 코드이므로 그대로 유지한다. 무해하고 의도를 명확히 전달한다.

  }

Fix 7 — 오프라인 Viewer 스레드 억제

7

System.cc — Viewer 생성 조건

Viewer(구 Pangolin)의 렌더링 스레드도 힙 활동을 발생시켜 비결정성에 기여할 수 있다. 오프라인 모드에서는 궤적 저장 후 Viewer 결과를 확인할 수 없으므로 스레드 자체를 생성하지 않음:

- if(bUseViewer)
+ // MODIFIED for "suppressing async viewer in offline SLAM mode to preserve trajectory determinism"
+ if(bUseViewer && !bOfflineMode)

Rerun 시각화는 Tracking 스레드에서 동기적으로 호출되므로 별도 억제 불필요. Viewer 억제와 무관하게 rerun 로그는 정상 기록된다.

규칙 5 — 진단 코드(fprintf 등) 추가 금지

비결정성 디버깅 시 fprintf 등의 진단 코드를 추가하면 힙 할당이 발생하여 비결정성의 위치 자체가 이동한다. 진단이 필요할 경우에는 반드시 모든 기존 진단 코드를 제거한 상태에서 측정해야 한다.
